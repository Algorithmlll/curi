Part 6: Vue Pages

Now that we have our configuration object ready to go, we can think about what our pages should look like. This tutorial will be rendering our website using Vue. If you prefer to use React, you should check out the Part 6: React Pages tutorial.

In this tutorial, we will be doing the following:

  • Modifying our Webpack configuration to support Vue.
  • Installing the @curi/vue package, learning about the CuriPlugin and one of the components it provides (<curi-link>).
  • Using the response object emitted by Curi to render our website.
  • Defining page components for each of the routes.
  • Adding links so that users can navigate between locations in the website.

If you are following along using the tutorial repo, please checkout the 06-pages-vue branch.

git checkout 06-pages-vue

Webpack#

Before we dive in, let's make sure that our build scripts can handle Vue. To do this, we just need to install Webpack's Vue loader and add support for .vue files to our Webpack configuration.

npm install --save-dev vue-loader vue-template-compiler
// webpack.config.js
const config = {
  // ...,
  resolve: {
    extensions: ['.js', '.vue'],
    alias: {
      vue: 'vue/dist/vue.js'
    }
  },
  module: {
    rules: [
      // ...
      {
        test: /.vue$/,
        loader: 'vue-loader'
      }
    ]
  }
};

The @curi/vue Package#

The @curi/vue package exports a Vue plugin, which provides components that know how to interact with Curi. For this tutorial, we will only be using one: <curi-link>. You can read more about the package in the @curi/vue documentation.

Installation#

Let's start by installing the @curi/vue package as well as Vue.

npm install @curi/vue vue

The Plugin#

The CuriPlugin exported by @curi/vue, should be registered with Vue after the Curi configuration object has been created. The plugin does a couple things. First, it will make your Curi configuration object and new responses/actions accessible to every component as through the this.$curi object. Second, it will register Curi specific components. For this tutorial, the only component that we care about is <curi-link>.

Instead of having to register the plugin manually (via Vue.use), @curi/vue also exports an installCuri function that will handle that for you. The install function will also subscribe to your Curi configuration object to handle new responses.

// index.js
import Vue from 'vue';
import { installCuri } from '@curi/vue';

const config = createConfig(history, routes);
installCuri(Vue, config);

The <curi-link> Component#

The <curi-link> component renders anchor (<a>) elements. However, unlike an anchor, we don't actually have to write the URI that we want to navigate to (the href). Instead, you use the to prop to pass the name of the route that you want to navigate to.

<curi-link to='Home'>Home</curi-link>
// <a href='/'>Home</a>

If the route that you are navigating to has any params, you pass them using the params prop.

// { name: 'Book', path: ':id' }
// (inherits 'books' path from parent route)
<curi-link to='Book' params={{ id: 1357 }}>
  Some Book
</curi-link>
// <a href='/books/1357'>Some Book</a>

If you need to pass any other location properties (query,hash, or state), you can provide them using the details prop.

<curi-link to='Contact' details={{ hash: 'email' }}>
  Contact by Email
</curi-link>
// <a href='/contact#email>Contact by Email</a>
Note: If you want to navigate outside of the application, use an anchor not a <curi-link>.
// interal
<curi-link to='Contact'>Contact</curi-link>
// external
<a href="https://github.com">GitHub</a>

The Response#

Being able to access the Curi configuration object is nice, but what we really need is to access the response objects that are emitted by Curi whenever the location changes. We could use the config.respond method that we covered in the configuration object tutorial, but installCuri takes care of that step for us. installCuri calls config.respond and in the response handler, it updates the reactive response and action properties of this.$curi whenever a new response is emitted.

While we do not have to manually subscribe to all responses, we do need to listen for the first response to be emitted so that we can render the application. The second argument to config.respond is an options object. If we pass the options { once: true }, then that function will only be called after the initial response is emitted.

Inside of the response handler function, we just need to render our root application component. All of our Curi related data will be available through this.$curi, so we don't have to attach any data to the Vue instance.

// index.js
import app from './components/app';

config.respond(response => {
  const vm = new Vue({
    el: '#root',
    // we'll get into the <app> soon
    template: '<app />',
    components: { app }
  });
}, { once: true });

The App#

At this point we have the ability to access our configuration object throughout our components and we are passing response objects to some <app> component that we haven't actually written yet. We should write those components now.

Our <app> component will be responsible for rendering our website based on the response object. To start, let's add a components directory to our src directory.

mkdir -p src/components

From up above, we know that our <app> will be receiving the response object as a prop. What should it render, though?

<!-- components/App.vue -->
<template>
</template>

<script>
  export default {
    name: 'app'
  };
</script>

Let's take a look at the properties of our response object to see which of its properties would be helpful for rendering our website.

{
 body: undefined,
 data: undefined,
 error: undefined,
 key: '1.0',
 location: { pathname: '/', ... },
 name: 'Home',
 params: {},
 partials: [],
 status: 200
}
Note: The Rendering with Responses guide goes into more detail about each of the properties of a response object.

In Part 3 of this tutorial, we added match.response functions that set the body property for each of our routes. There, we just used a placeholder string, but now we can actually set the component for each route. Instead of returning a string, what if set.body set the body to be a Vue component? Then, our render function can use the body property of our response object to render our website.

<!-- components/App.vue -->
<template>
  <component :is="$curi.response.body" />
</template>

<script>
export default {
  name: 'app'
};
</script>

We'll expand on that later on, but for now, let's go ahead and define the components for each of our routes.

The Route Components#

To refresh your memory, we have "Home", "Contact", "Book List", "Book", "Checkout", and "Not Found" pages that we will need to create components for. We can write some barebones components and add some more content later on.

<!-- components/Home.vue -->
<template>
  <div class='home'>
    Welcome to our book store!
  </div>
</template>
<!-- components/Contact.vue -->
<template>
  <div class='contact'>
    You can contact us by fax at 1-206-555-0123.
  </div>
</template>
<!-- components/BookList.vue -->
<template>
  <div class='book-list'>
    Available Books
  </div>
</template>
<!-- components/Book.vue -->
<template>
  <div class='book'>
    Book
  </div>
</template>
<!-- components/Checkout.vue -->
<template>
  <div class='checkout'>
    Checkout
  </div>
</template>
<!-- components/NotFound.vue -->
<template>
  <div class='not-found'>
    Page not found
  </div>
</template>

All of these components should be imported in our routes.js and set using set.body in their respective match.response functions.

// routes.js
import Home from './components/Home';
import Contact from './components/Contact';
import BookList from './components/BookList';
import Book from './components/Book';
import Checkout from './components/Checkout';
import NotFound from './components/NotFound';

const routes = [
  {
    name: 'Home',
    path: '',
    match: {
      response: ({ set }) => {
        set.body(Home);
      }
    }
  },
  {
    name: 'Contact',
    path: 'contact',
    match: {
      response: ({ set }) => {
        set.body(Contact);
      }
    }
  },
  {
    name: 'Checkout',
    path: 'checkout',
    match: {
      response: ({ set }) => {
        set.body(Checkout);
      }
    }
  },
  {
    name: 'Book List',
    path: 'books',
    match: {
      response: ({ set }) => {
        set.body(BookList);
      }
    }
    children: [
      {
        name: 'Book',
        path: ':id',
        match: {
          response: ({ set }) => {
            set.body(Book);
          }
        }
      }
    ]
  },
  {
    name: 'Not Found',
    path: '(.*)',
    match: {
      response: ({ set }) => {
        set.body(NotFound);
      }
    }
  }
];

export default routes;

Now, if we load up our application, we will render our home page. Unfortunately, there is no way to navigate to any of our other pages. We will need to add some <curi-link>s to our application.

A Navigation Menu#

We can write a simple <NavLinks> menu component to add navigation to our application. From this menu, we only need to be able to navigate to our "Home", "Contact", "Book List", and "Checkout" routes. Navigation to individual books will be done from the book list page.

We will use a <nav> element as the parent for our <NavLinks>. Inside of that is a <ul> and then each of our routes will be <Link>s wrapped in <li>s.

<!-- components/NavLinks.vue -->
<template>
  <nav>
    <ul>
      <li>
        <curi-link to='Home'>Home</curi-link>
      </li>
      <li>
        <curi-link to='Contact'>Contact Us</curi-link>
      </li>
      <li>
        <curi-link to='Book List'>Books for Sale</curi-link>
      </li>
      <li>
        <curi-link to='Checkout'>Checkout</curi-link>
      </li>
    </ul>
  </nav>
</template>

Let's import and render that the <NavLinks> in our <app>. We'll also add some wrapper elements to keep our content organized.

<!-- components/App.vue -->
<template>
  <div>
    <header>
      <NavLinks />
    </header>
    <main>
      <component :is="$curi.response.body" />
    </main>
  </div>
</template>

<script>
  import NavLinks from './NavLinks';
  export default {
    name: 'app',
    components: { NavLinks }
  };
</script>

At this point, we can navigate between most of our routes. However, we still need to add navigation to our books.

Navigating to Our Books#

Our "Book" route is different than all of our other routes because the book path includes an id param. This means that we need to actually have "id" values to pass to our <curi-link>s. Later on, we'll generate some better data, but for now we can just generate a placeholder list in a module called books.js.

 // books.js
const books = [
  { id: 0 },
  { id: 1 },
  { id: 2 },
  { id: 3 }
];
export default books;

Then, in <BookList> we can iterate over this list to generate links to our books.

<!-- components/BookList.vue -->
<template>
  <div class='book-list'>
    <h1>Available Books</h1>
    <div class='books'>
      <div v-for="book in books" :key="book.id" class='book-item'>
        <curi-link to='Book' :params="{ id: book.id }">
          Book {{book.id}}
        </curi-link>
      </div>
    </div>
  </div>
</template>

<script>
  import books from '../books';
  export default {
    data: function() {
      return { books };
    }
  };
</script>

Book Props#

The <curi-link>s above allow us to navigate to our books, but we don't actually have any information about which book we are supposed to be seeing. It would really help if our <Book> was able to access the parsed params so that it can render the information for the correct book.

The params object is a property of our response object. We can access the response in our components using this.$curi.response, so we don't actually have to manually pass it as a prop.

<!-- components/App.vue -->
<template>
  <div>
    <header>
      <NavLinks />
    </header>
    <main>
      <component :is="$curi.response.body" />
    </main>
  </div>
</template>

<script>
  import NavLinks from './NavLinks';

  export default {
    name: 'app',
    props: ['response'],
    components: { NavLinks }
  };
</script>

Next, we just need to update our <Book> component to access the params object and figure out which book to render content for.

<!-- components/Book.vue -->
<template>
  <div class='book'>
    Book {{$curi.response.params.id}}
  </div>
</template>
